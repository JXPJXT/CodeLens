Section 1

Project Overview

CodeLens AI is an AI-powered code review assistant that works like a debugger for code quality — not for runtime errors, but for bugs, security vulnerabilities, and bad coding patterns. A developer writes code in VS Code, the tool instantly highlights risky lines using a deep learning model, and on demand explains the issue and suggests a fix using a large language model.

The core idea is a two-layer system: a fast, locally-running neural network that detects suspicious patterns in milliseconds, paired with a powerful cloud-based language model that explains what the problem is and how to fix it in plain English.

⚠️ Problem Statement

Developers spend enormous time on code review — finding bugs, security flaws, and bad practices. Existing tools are either too rigid (static analyzers that flag everything and explain nothing) or too slow (asking an AI chatbot means switching context). There is no tool that works inside the editor, in real time, that both detects a bug and explains it.

✅ Our Solution

CodeLens behaves exactly like a debugger — but for code quality instead of runtime state:

  1. Developer saves a file in VS Code
  2. Extension sends code to the backend API
  3. CNN model runs in under 100ms and returns a risk score + which lines are suspicious
  4. Risky lines are highlighted red/yellow in the editor
  5. Developer clicks 'Explain' → Llama 3.1 8B generates a plain-English explanation and fix suggestion
  6. Feedback (correct / wrong) is stored for future improvement

🎓 Viva Defense — Section 1

Q: Why not just use ChatGPT or Copilot directly?
A: ChatGPT requires context switching — leaving the editor. CodeLens integrates directly into VS Code, runs analysis on save in <50ms, and combines a fast local CNN for detection with an LLM only when the developer explicitly asks for an explanation. This two-speed design gives real-time feedback without API latency on every keystroke.
Q: What makes this different from static analysis tools like SonarQube?
A: Static analyzers use handwritten rules — they can only catch what someone explicitly programmed. Our CNN learns patterns from 27,000 real vulnerable functions, so it can catch patterns no one wrote a rule for. Plus, the LLM explains *why* something is risky in plain English, which static tools never do.
Q: Is this an original project or a wrapper around existing tools?
A: The CNN model is trained from scratch on the Devign dataset. The interpretability pipeline (activation mapping → line-level scores) is custom. The LLM integration uses prompt engineering we designed and evaluated with BLEU-4 scores. The VS Code extension, FastAPI backend, and routing logic are all original code.

🎓 Viva Defense — Section 2

Q: Why FastAPI instead of Flask or Django?
A: FastAPI has native async support (crucial for concurrent LLM API calls), automatic OpenAPI documentation, and Pydantic validation. Flask would work but lacks async. Django is too heavyweight — we don't need an ORM, admin panel, or template engine for an API service.
Q: Why separate the CNN and LLM into different services?
A: Separation of concerns. The CNN runs on every file save (<50ms) — it must be fast. The LLM runs only on explicit user request (1-3s). Combining them would mean either slowing down every save or wasting LLM calls. The two-speed architecture matches how developers actually work.
Q: What happens if the CNN model file is missing or corrupted?
A: Graceful degradation. The cnn_service catches the FileNotFoundError at import time and returns a safe fallback response with risk_score=0. The system continues working — the heuristic analyzer and LLM still function. The server never crashes.
Q: Why a VS Code extension and not a web app?
A: Developers write code in editors, not browsers. A web app would require copy-pasting code — breaking the workflow. A VS Code extension integrates into the exact environment where code is written, providing inline highlighting on the actual source file.
Section 3

Deep Learning Component — TextCNN Bug Classifier

This is the core intelligence of CodeLens for C code. It is a neural network trained to look at a C function and answer one question: is this code buggy or clean?

3.1 What is Deep Learning?

Think of deep learning as teaching a computer to recognise patterns by showing it thousands of examples. Just like a child learns to recognise a cat by seeing hundreds of pictures of cats, our neural network learns to recognise buggy code by seeing thousands of examples of buggy and clean functions.

A neural network is made up of layers. Each layer transforms the data slightly, extracting more and more abstract features. By the end, the network has learned which patterns indicate a bug.

Analogy: How the CNN thinks about code

Imagine you are reading a document and you are looking for signs of fraud. You might scan for specific phrases: 'wire transfer', 'urgent', 'confidential account'. You do not read every word — you look for suspicious local patterns.

Our CNN does exactly this, but for code. It scans for suspicious code patterns like:

  • strcpy(dest, src) — no bounds check, classic buffer overflow
  • ptr->field — accessing a pointer field without checking if ptr is NULL
  • malloc(size) — allocating memory without checking if it returned NULL

3.2 The Dataset — Devign

Dataset: DetectVul/devign from the CodeXGLUE benchmark on HuggingFace.

PropertyDetail

Each entry in the dataset is a complete C function (sometimes dozens of lines long) paired with a binary label: 1 for buggy, 0 for clean. The bugs are real — they were discovered and patched in actual software, so the labels are trustworthy.

3.3 Converting Code to Numbers — Tokenization and Embeddings

Neural networks cannot understand text. They only work with numbers. The pipeline to convert code into numbers has two steps.

Step 1: Tokenization

Tokenization is the process of breaking text into small units called tokens. We use BPE (Byte Pair Encoding), the same technique used by GPT models.

BPE is smart about code. It recognises that a long identifier like getUserData is better split into [get, User, Data] than treated as one unknown word. This matters because many bug-related function names share sub-parts.

Example tokenization of a C snippet:

Input: char *ptr = malloc(size); if(ptr) { strcpy(ptr, input); }

Output: ["char", " *", "ptr", " =", " malloc", "(", "size", ")", ";", " if", "(", "ptr", ")", " {", " str", "cpy", ...]

Step 2: Embeddings

Embeddings convert each token ID into a vector of floating-point numbers. Think of it as a coordinate in a high-dimensional space where similar tokens are placed near each other.

For example, in a well-trained embedding space: strcpy and strncpy would be close to each other, and both would be near memcpy — because they are all memory copy operations. The model learns these relationships automatically during training.

Each code function becomes a 2D matrix: rows are tokens, columns are embedding values. This matrix is what the CNN actually processes.

Embedding matrix shape:

Function → 256 tokens × 128 embedding dimensions = matrix of shape (256, 128)

3.4 The TextCNN Architecture — What It Is and Why We Used It

CNN stands for Convolutional Neural Network. It was originally invented for image recognition (detecting patterns in pixels), but researchers discovered it works extremely well for text too — because both images and text have local patterns that matter.

Why CNN for code and not an RNN or Transformer?

The key insight: most bugs are local.

A buffer overflow happens in 2–3 lines. A null pointer dereference is visible in a single expression. These patterns do not require understanding the entire function — they require detecting suspicious short sequences.

CNN advantages:

  • Fast: Convolutions run in parallel — inference takes under 50ms, making real-time analysis possible
  • Explainable: Filter activations map back to specific tokens, showing which code triggered detection
  • Stable: No vanishing gradient problems that plague RNNs on long sequences

The TextCNN Architecture in Detail

Our TextCNN has four stages:

Stage 1: Embedding Layer

Input: token IDs of shape (batch_size, 256). Output: embedding matrix of shape (batch_size, 256, 128). The embedding layer is a lookup table — each token ID maps to a 128-dimensional vector.

Stage 2: Three Parallel Convolutional Branches

This is the heart of the model. Three Conv1D layers run simultaneously, each with a different filter size:

BranchWindowDetectsStrength
Filter size 3Looks at 3 tokens at a timeCatches: null pointer, array boundsMost sensitive to micro-patterns
Filter size 4Looks at 4 tokens at a timeCatches: unsafe function callsBalanced coverage
Filter size 5Looks at 5 tokens at a timeCatches: multi-step unsafe patternsBest for longer idioms

Each branch produces 128 feature maps. Think of each feature map as a detector for one specific pattern. After convolution, max-over-time pooling extracts the strongest signal from each detector across the whole function — regardless of where in the code it appears.

Stage 3: Regularization

BatchNorm1D normalizes the 384-dimensional combined output (3 branches × 128 filters) so values are on a consistent scale. This speeds up training significantly.

Dropout (p=0.5) randomly sets half the neurons to zero during training. This forces the model to learn redundant representations and prevents overfitting — the most common failure mode in deep learning where a model memorizes training data instead of learning generalizable patterns.

Stage 4: Classification Head

A single Linear(384, 1) layer produces one number. When passed through Sigmoid during inference, this becomes a probability between 0 and 1. Above the threshold → buggy. Below → clean.

🎓 Viva Defense — Section 3

Q: Why TextCNN and not a Transformer like BERT for the real-time model?
A: Speed. TextCNN runs in <50ms — fast enough for on-save analysis. A transformer like BERT takes 200-500ms per inference and requires 6GB+ VRAM. For real-time highlighting, CNN is the right trade-off. We use CodeBERT separately for deep analysis on demand.
Q: Why three filter sizes (3, 4, 5)? Why not just one?
A: Different bug patterns span different token lengths. A null check is 2-3 tokens, an unsafe function call is 4 tokens, a multi-step pattern like malloc-without-null-check spans 5+ tokens. Multiple filter sizes capture all these patterns simultaneously. This is the standard TextCNN design from Yoon Kim (2014).
Q: Why 128 embedding dimensions and not 256 or 768?
A: 128 is the sweet spot for our dataset size (~27k functions). Higher dimensions would overfit — the model would memorize rather than generalize. Our experiment with 768-dim CodeBERT embeddings (Run 4) confirmed that higher dimensions didn't improve AUC.
Q: What is max-over-time pooling and why use it?
A: It takes the maximum activation from each filter across all positions. This means the model detects a pattern regardless of *where* in the function it appears — line 3 or line 30. It also produces a fixed-size output (128 per branch) regardless of input length.
Q: Why use BPE tokenization instead of character-level or word-level?
A: Character-level loses semantic meaning. Word-level marks every unique identifier as unknown. BPE splits getUserData into [get, User, Data] — preserving semantics while handling never-seen-before identifiers. It's the same tokenization GPT and CodeBERT use.
Section 4

Training — How the Model Learned

4.1 The Training Setup

Training is the process of showing the model thousands of examples and adjusting its internal parameters (weights) until it gets good at predicting the right label.

ComponentValue / SettingPurpose
Loss FunctionBCEWithLogitsLossMeasures how wrong each prediction is
OptimizerAdam (lr=1e-3)Adjusts weights to reduce loss
Batch Size64How many functions processed at once
EpochsUp to 20Full passes through training data
Early StoppingPatience = 3–5 epochsStops when validation stops improving
SchedulerReduceLROnPlateauHalves learning rate when stuck

Understanding BCEWithLogitsLoss

BCE stands for Binary Cross-Entropy. For each prediction, it measures: how surprised was the model by the correct answer? If the model predicted 0.9 buggy and the label is 1 (buggy), the loss is very low. If the model predicted 0.1 buggy but the label is 1, the loss is very high. The optimizer then adjusts the weights to reduce this surprise.

Understanding Early Stopping

We monitor validation loss after each epoch. If it stops improving for 3–5 consecutive epochs, we stop training and restore the best model. This is crucial because training loss always keeps dropping (the model memorizes training data), but validation loss eventually rises — indicating overfitting.

Overfitting example from our training:

Epoch 5: Train Loss: 0.51 | Val Loss: 0.58 | Val Acc: 0.68 ← best

Epoch 6: Train Loss: 0.49 | Val Loss: 0.62 | Val Acc: 0.66 ← overfitting starts

Epoch 7: Train Loss: 0.46 | Val Loss: 0.65 | Val Acc: 0.65 ← definitely overfitting

4.2 Class Imbalance and Weighted Loss

Devign has roughly 55% clean and 45% buggy functions. This imbalance causes the model to take the lazy shortcut of predicting 'clean' whenever it is uncertain — which is wrong behaviour for a bug detector.

We addressed this by computing a class weight and passing it to the loss function. The sqrt dampens the weight so it nudges without overcorrecting:

num_buggy  = count of buggy functions in training set
num_clean  = count of clean functions in training set
pos_weight = sqrt(num_clean / num_buggy)

# This tells the loss function: treat each buggy sample
# as if it appeared pos_weight times in the dataset
criterion = BCEWithLogitsLoss(pos_weight=pos_weight)

🎓 Viva Defense — Section 4

Q: Why BCEWithLogitsLoss instead of CrossEntropyLoss?
A: BCEWithLogitsLoss is designed for binary classification — it combines sigmoid and BCE in one numerically stable operation. CrossEntropyLoss is for multi-class problems. Since we have exactly two classes (buggy/clean), BCE is the correct choice.
Q: What is early stopping and why is it critical?
A: Early stopping monitors validation loss and stops training when it stops improving. Without it, the model keeps memorizing training data (overfitting). In our experiments, validation loss started rising after epoch 5-6 while training loss kept dropping — classic overfitting that early stopping prevents.
Q: Why sqrt dampening for pos_weight instead of the full ratio?
A: Run 2 showed that the full ratio (1.22) overcorrected — bug recall jumped but created 579 false positives. The sqrt (≈1.10) provides a gentler nudge. This is a standard technique in imbalanced classification to avoid the model swinging too far toward the minority class.
Q: Why Adam optimizer and not SGD?
A: Adam adapts the learning rate per-parameter, which is essential for embedding layers where different tokens update at different rates. SGD with a fixed learning rate struggles with the sparse gradients from embeddings. Adam converges faster and more reliably for NLP tasks.
Section 5

All Training Approaches — What We Tried and What Happened

We ran six distinct experiments. Each is documented here with the exact change made, the results, and the analysis of why it performed the way it did.

Run 1 — Baseline TextCNN with Hash Tokenizer

What we did: Simple TextCNN with hash-based tokenization (hash(token) % 50000), no class weighting, 10 epochs, Adam lr=1e-3

Key flaw: hash(token) % 50000 is lossy. Different tokens can hash to the same ID (collision). The model received corrupted input but still learned something.

Results: AUC: 0.722 | F1: 0.586 | Bug recall: 53.9% | Clean recall: 74.6%

Analysis: This became our best AUC result across all TextCNN experiments — a sobering indicator that the architecture ceiling, not the tokenizer, was the main constraint. The model was strongly biased toward predicting 'clean', missing 46% of actual bugs.

Run 2 — Weighted Loss (Full Ratio)

What we did: Switched to BCEWithLogitsLoss with pos_weight = num_clean / num_buggy (full ratio, approximately 1.22)

Results: AUC: 0.707 | F1: 0.617 | Bug recall: 65.2% | Clean recall: 60.8%

Analysis: The weighting overcorrected. Bug recall jumped from 53.9% to 65.2% — the model started catching more real bugs. But it also started flagging too many clean functions as buggy (579 false positives vs 375 before). AUC dropped because the model's overall discriminative ability did not improve — it just moved its decision boundary. F1 improved because we care about bugs more than clean predictions.

Run 3 — Weighted Loss (Square Root Dampening)

What we did: Changed pos_weight = sqrt(num_clean / num_buggy) to soften the correction

Results: AUC: 0.702 | F1: 0.587 | Bug recall: 56.1% | Clean recall: 70.1%

Analysis: The model swung back toward clean predictions but did not fully recover. AUC continued to slide. This suggested that tuning the loss function alone was moving the threshold, not improving the model's actual ability to distinguish buggy from clean code.

Run 4 — Frozen CodeBERT Embeddings + Projection Layer

What we did: Replaced random 128-dim embeddings with pretrained CodeBERT embeddings (768-dim, frozen). Added a learnable projection layer: Linear(768, 128) + ReLU to compress before the CNN.

Results: AUC: 0.708 | F1: 0.608 | Most balanced confusion matrix of all runs

Analysis: The confusion matrix became the most symmetric: 992 correct clean, 485 wrong; 495 wrong, 760 correct buggy. The model was no longer strongly biased in either direction. However, AUC did not improve. The frozen CodeBERT embeddings, even with the projection layer, could not give the CNN richer information — because the CNN filters could not fine-tune the representation to the bug detection task.

Run 5 — Unfrozen CodeBERT Embeddings (Regression)

What we did: Set freeze=False to allow the embeddings to update during training. Used differential learning rates: 2e-5 for embeddings, 3e-4 for CNN layers. Increased to 20 epochs with patience=5.

Results: AUC: 0.692 | F1: 0.602 ← worst result

Analysis: Unfreezing caused catastrophic interference. The randomly initialized projection layer produced large gradients that propagated back through the embedding layer and destroyed the pretrained structure before it could be useful. This run confirmed the TextCNN architecture is incompatible with full embedding fine-tuning at this scale without a careful warmup strategy.

Run 6 — Clean Reset (Confirmed Baseline)

What we did: Reverted to the simple TextCNN with no CodeBERT dependencies, plain Adam, no weighting, but added proper early stopping (patience=3) and optimal threshold tuning via ROC curve.

Results: AUC: 0.700–0.714 | Optimal threshold: 0.346 | Bug recall: 74.6%

Analysis: The optimal threshold of 0.346 (found by maximising tpr-fpr on the ROC curve) produced bug recall of 74.6% — the highest of all runs. This is appropriate for a code review tool: it is better to flag too many things and let the developer decide than to silently miss real bugs. This became our locked-in TextCNN result.

🎓 Viva Defense — Section 5

Q: Your best AUC is only 0.714 — isn't that low?
A: For TextCNN on Devign, published results consistently report 0.70-0.73 AUC. Our 0.714 is within the expected range. The ceiling is the architecture, not our implementation. State-of-the-art models (graph neural networks) reach 0.84 but require code's control flow graph — a fundamentally different input representation.
Q: Why did unfreezing CodeBERT embeddings make things worse?
A: Catastrophic interference. The randomly initialized projection layer (Linear 768→128) generated large, random gradients that flowed back into the pretrained embeddings and destroyed their learned structure. A warmup strategy (freezing embeddings for N epochs, then unfreezing with very low LR) would fix this, but wasn't worth the complexity for a CNN-based model.
Q: What is the optimal threshold and how did you find it?
A: The default threshold is 0.5, but for a bug detector, missing a real bug (false negative) is worse than a false alarm. We plotted the ROC curve and found the threshold that maximizes TPR - FPR, which was 0.346. This gives 74.6% bug recall — catching 3 out of 4 real bugs.
Section 6

CodeBERT Fine-Tuning — The Deep Analysis Model

After exhausting TextCNN's ceiling (~0.72 AUC), we introduced a second, more powerful model: fine-tuned CodeBERT. This is not a replacement for the TextCNN — it is a complement. TextCNN handles fast real-time scanning; CodeBERT handles deep on-demand analysis.

6.1 What is CodeBERT?

CodeBERT is a large pretrained transformer model developed by Microsoft, trained on 8.35 million code-comment pairs across 6 programming languages. It uses the BERT architecture — a 12-layer deep transformer with 125 million parameters.

The key advantage of CodeBERT for our task is that it was pretrained to understand the relationship between code and natural language — which is exactly what a code review tool needs.

What does 'fine-tuning' mean?

CodeBERT has already learned, from 8 million examples, what code looks like and how it relates to natural language. Fine-tuning means we take this pretrained model and continue training it on our specific task (Devign bug detection) for just a few more epochs.

The analogy: CodeBERT is a software engineer with 5 years of general coding experience. Fine-tuning is giving that engineer 2 weeks of intensive training specifically on security vulnerability detection. They do not forget everything they knew — they become specialized.

The [CLS] token: BERT-style models prepend a special [CLS] (classification) token to every input. 🎓 Viva Defense — Section 6

Q: What is the difference between using CodeBERT embeddings in TextCNN vs fine-tuning CodeBERT directly?
A: In TextCNN, we only used CodeBERT's embedding layer (a lookup table) and froze it — the CNN filters processed these static embeddings. Fine-tuning CodeBERT means using the entire 12-layer transformer encoder and updating all 125M parameters on our task. The transformer's self-attention captures long-range dependencies that CNN filters cannot.
Q: Why is the batch size only 8 for CodeBERT?
A: Memory constraint. CodeBERT has 125M parameters. With 128-token sequences and float32, each batch element uses ~200MB of GPU memory for activations + gradients. Batch size 8 × 200MB = 1.6GB, leaving enough room on a 6GB RTX 3050. Batch size 16 would cause CUDA out-of-memory.
Q: What is the [CLS] token and why do you use only that for classification?
A: BERT-style models prepend a special [CLS] token. Through 12 layers of self-attention, this token attends to every other token and accumulates a summary of the entire sequence. Using its 768-dim output for classification is standard practice — it was specifically designed for this purpose during BERT's pre-training.
Section 7

Interpretability — Why a Line is Flagged

One of the most important features of CodeLens is that it does not just say 'this code is buggy' — it shows which specific lines triggered the detection. This is called interpretability, and it is what separates a useful tool from a black box.

7.1 CNN Activation Mapping

The interpretability pipeline works by tracing back from the model's decision to the specific tokens that caused it:

  1. Run the forward pass through the CNN and capture the raw feature maps before max-pooling (shape: batch × filters × positions)
  2. For each filter, record which position in the sequence had the maximum activation — this is what max-pooling selected
  3. Map that position back to a token in the original tokenized sequence
  4. Map that token back to a line number in the original source code
  5. Score each line by the average activation strength of filters that pointed to it

Example output:

Line 12: risk_score=0.91 → strcpy(dest, input) — filter for 'unsafe_string_copy' activated

Line 7: risk_score=0.63 → ptr = malloc(len) — filter for 'unchecked_alloc' activated

Line 3: risk_score=0.21 → int x = 0 — low activation, safe

7.2 Heatmap Visualization

The interpretability module includes a terminal heatmap using colorama that prints each line with a colour-coded risk score — red for high risk, yellow for medium, green for low. This directly satisfies the DL objective requirement for filter activation visualization.

🎓 Viva Defense — Section 7

Q: How exactly do you map from CNN filter activations back to source code lines?
A: Max-pooling selects the position with the highest activation per filter. That position corresponds to a token index in the tokenized sequence. We maintain a mapping from token indices → original character offsets → line numbers. Each line's risk score is the average activation strength of all filters that pointed to tokens on that line.
Q: Why is interpretability important for this tool?
A: A tool that says "this function is buggy" without pointing to the specific line is useless — the developer still has to read the entire function. Line-level highlighting turns a classification tool into a debugging tool. It also builds trust: developers can verify whether the flagged line actually looks suspicious.
Q: Is this the same as Grad-CAM or SHAP?
A: No. Grad-CAM uses gradient information and is designed for image CNNs. SHAP is model-agnostic but very slow. Our approach uses the CNN's own max-pooling indices — it's nearly free computationally because max-pooling already records which position won. It's specific to TextCNN's architecture but much faster than gradient-based methods.

Token Fertility

Token fertility = number of BPE tokens produced / number of whitespace words in the original. A fertility of 1.0 means perfect 1:1 mapping. Higher fertility means BPE is splitting more aggressively.

For code, BPE fertility is typically 1.4–1.8 for C and 1.6–2.1 for Python. This is because Python uses longer, more expressive identifiers and more complex string patterns.

Why BPE Wins for Code

  • Handles unknown identifiers: Word tokenization marks every unique function name as <UNK> (unknown). BPE breaks it into recognizable pieces
  • Preserves semantics: maxRetryCount splits into [max, Retry, Count] — the model understands each component
  • Smaller vocabulary: 50,000 BPE tokens can cover billions of unique identifier combinations
  • Works across languages: BPE trained on code handles Python, C, and JS without language-specific rules

🎓 Viva Defense — Section 8

Q: What is token fertility and why does it matter?
A: Token fertility = BPE tokens / whitespace words. Higher fertility means more splitting. For code it's 1.4–1.8 for C and 1.6–2.1 for Python. It matters because higher fertility increases sequence length — if our max_len is 256, a function that's 200 whitespace words becomes 320+ BPE tokens, getting truncated. Understanding fertility helps us set the right max_len.
Q: Why compare BPE and word tokenization specifically?
A: They represent two extremes. Word tokenization treats every unique identifier as unknown () — losing all semantic information. BPE finds a middle ground by splitting into subword units. Comparing them quantifies exactly how much semantic information BPE preserves, which directly explains why the model can generalize to unseen code.
Q: How does this NLP analysis connect to your deep learning model?
A: The tokenizer is the model's eyes. If tokenization is bad, the model sees corrupted input. Our Run 1 (hash tokenizer with collisions) vs Run 6 (proper BPE) comparison proves this — better tokenization directly improved model stability. The NLP analysis validates our tokenization choice with quantitative evidence.

Template 3: Few-Shot (3 Examples)

Three diverse examples covering different bug categories. This is the most effective template — it gives the model a pattern to follow and shows the expected level of technical depth.

Why few-shot works best: The model is shown what good looks like before being asked to produce it. Three examples covering different bug types (memory, null pointer, overflow) condition the model to consider all these categories for every new input.

🎓 Viva Defense — Section 9

Q: Why Llama 3.1 8B and not GPT-4 or a larger model?
A: GPT-4 is proprietary, expensive, and has strict rate limits. Llama 3.1 8B is open-source, available for free via Groq's free tier, and performs comparably on code review tasks. Using an open model also means we could self-host it in the future, removing API dependency entirely.
Q: Why does few-shot prompting work better than zero-shot?
A: Few-shot gives the model concrete examples of what a good code review looks like — the format, the depth, the technical level. Without examples (zero-shot), the model might give a generic, shallow response. With 3 diverse examples covering memory bugs, null pointers, and overflows, it learns to check for all these categories systematically.
Q: What is BLEU-4 and is it a good metric for code reviews?
A: BLEU-4 measures 4-gram overlap between generated and reference text. It's a standard NLP metric but not ideal for code reviews — two completely different wordings can both be correct reviews. That's why we supplement with manual relevance, clarity, and correctness scores (1–3 scale). BLEU satisfies the academic requirement; manual scoring provides real quality assessment.
Q: What happens if the Groq API is down or rate-limited?
A: The llm_service catches all network exceptions and returns a fallback string: "LLM review unavailable. Please try again later." The extension displays this gracefully. The CNN detection still works — only the explanation feature degrades. This is the graceful degradation pattern used throughout the system.

🎓 Viva Defense — Section 10

Q: Why load the CNN model at import time instead of per-request?
A: Model loading takes 1–2 seconds (reading from disk, initializing tensors). Doing this per-request would add 1–2s latency on every /analyze call. Loading once at startup means subsequent requests only pay the inference cost (~50ms). This is the singleton pattern — standard practice for ML model serving.
Q: Why pydantic-settings for configuration instead of just os.environ?
A: Pydantic-settings provides type validation (ensuring RISK_THRESHOLD is a float, not a typo string), default values, and centralization. With os.environ scattered across files, changing a variable name means hunting through the entire codebase. With BaseSettings, all config is in one file with documented types and defaults.
Q: Why SQLite for the feedback database and not PostgreSQL?
A: SQLite requires zero setup — no server, no installation, no configuration. For a single-user development tool, it handles the load perfectly. If this were deployed for a team of 50 developers, we'd migrate to PostgreSQL. The DB layer is abstracted, so the switch would only require changing the connection string.
Section 11

System Design Concepts

11.1 Separation of Concerns

Each service has exactly one job. CNN service does inference. LLM service calls Groq. Cache service stores responses. Router service decides which path to take. This means any component can be replaced or upgraded without touching the others.

11.2 Graceful Degradation

The system never breaks. If the CNN model file is missing, cnn_service returns a safe fallback. If Groq is rate-limited, llm_service returns an informative fallback string. If the backend is unreachable, the extension logs to the output channel and continues working.

11.3 Caching

LLM API calls are expensive and slow. The cache layer stores responses keyed by an MD5 hash of the code and language. Repeated analysis of the same function returns instantly from cache. This makes the system behave well during live demonstrations.

11.4 Environment-Based Configuration

Every secret (Groq API key) and every tunable value (risk threshold, model path) lives in .env and is loaded through pydantic-settings. No hardcoded secrets anywhere in source code. Deploying to a different environment or swapping to a different LLM requires only editing one file.

11.5 Feedback Loop

Every developer thumbs-up or thumbs-down is stored in SQLite with the code and prediction. This is the data foundation for future model improvement — you can retrain the CNN on cases where it was wrong, making the system improve over time from real usage.

🎓 Viva Defense — Section 11

Q: What design patterns did you use and why?
A: (1) Strategy Pattern in the language router — different analysis strategies per language. (2) Singleton for model loading — one instance, reused across requests. (3) Cache-Aside for LLM responses — check cache first, call API only on miss. (4) Facade pattern — the FastAPI gateway hides the complexity of CNN, analyzer, and LLM services behind simple /analyze and /review endpoints.
Q: How would you scale this for production use?
A: Replace SQLite with PostgreSQL, swap in-memory cache with Redis, containerize with Docker, deploy behind nginx. For heavy load: run multiple FastAPI workers with Gunicorn, serve the CNN on a GPU instance, and add request queuing for LLM calls. The current architecture supports all of these changes without restructuring.
Q: Why is the feedback loop important if you already have a trained model?
A: The model was trained on C functions from FFmpeg/QEMU. Real-world code looks different. Collecting developer feedback (correct/wrong) creates a domain-specific dataset for retraining. Over time, the model improves on the specific codebase it's deployed on — this is the core principle of online learning and continuous improvement.
